## Goal

Produce **one canonical proxy address shared across Mainnet / Base / Arbitrum / Optimism**, replacing the live v1.1.0 deploys on Base + OP with a fresh v1.2.0 deploy. Wipes state on Base + OP (no posts/karma/whitelisters carry over) and brings Mainnet + Arb online for the first time.

## Why a salt bump (operator decision)

Contract code is **identical** to live Base/OP v1.1.0 (`git diff be695f9 HEAD -- contracts/` is empty). The salt bump is intentional state-wipe + clean v1.2.0 marker, not a code change.

Trade-off accepted by the operator:

- ✅ Fresh state across all 4 chains, single canonical launch event.
- ❌ Old Base/OP proxy `0x585192Be5805Dc6D2F326369E6D0F8B7E11a7974` becomes orphaned. Anyone bookmarked against it must re-point. Indexer / mesh / frontend / notifier configs all need address swap.

## Cross-chain identical-address invariants

Every input below MUST be byte-identical on every chain or the proxy address diverges.

| Input | Value |
|---|---|
| `IMPL_SALT` | `keccak256("thatsRekt.impl.v1.2.0")` |
| `UPGRADE_TIMELOCK_SALT` | `keccak256("thatsRekt.upgradeTimelock.v3")` |
| `ADD_TIMELOCK_SALT` | `keccak256("thatsRekt.addTimelock.v3")` |
| `PURGE_TIMELOCK_SALT` | `keccak256("thatsRekt.purgeTimelock.v3")` |
| `PROXY_SALT` | `keccak256("thatsRekt.proxy.v3")` |
| `GOVERNANCE_OWNER` | `0x59E4DBc95BD312A882Bb36b7f3E8298682340679` (thatsRekt Safe multisig — verified deployed with identical bytecode on Mainnet, Base, Arb, OP) |
| `WHITELIST_OPERATOR` | `0xda1b9dFA299d655135C1ECdc4f0b4c9aED9a7f45` (bauti.eth — operator decision 2026-05-07, swap from cold wallet) |
| `PURGE_REMOVER_EOA` | `0xda1b9dFA299d655135C1ECdc4f0b4c9aED9a7f45` (bauti.eth — same swap) |
| `INITIAL_WHITELISTERS` (exact order) | `0x5822B262EDdA82d2C6A436b598Ff96fA9AB894c4,0xda1b9dFA299d655135C1ECdc4f0b4c9aED9a7f45,0x9E8680dbBcA1127add812abe209A10E621b385dF,0x24C2167054A9A9e00F67233F1eBc4060501f54FA,0xE0396d6d738e726D39f96099b8f6a55d11184374,0xb5A6c8ca369e38050784e2A6793beE6447109340` |
| Deployer (broadcaster) EOA | `0xb5A6c8ca369e38050784e2A6793beE6447109340` (DAMM hot wallet — also pre-whitelisted) |
| CREATE2 factory | `0x4e59b44847b379578588920cA78FbF26c0B4956C` (Arachnid singleton) |

**Why each one matters:**
- Salts change → different `keccak256` input to CREATE2 → different address.
- `GOVERNANCE_OWNER` is encoded into both the upgrade-TLC and add-TLC initcode (proposer arg). Different value → different TLC address → different proxy initcode (which references TLCs in `initialize` calldata) → different proxy address.
- `WHITELIST_OPERATOR` is encoded as the proxy's `whitelistRemover` slot AND as the canceller of upgrade + add TLCs. Different value → different initcode at proxy + 2 of 3 TLCs.
- `PURGE_REMOVER_EOA` is encoded into purge-TLC initcode (proposer arg) AND into proxy `initialize` calldata. Different value → different purge TLC + proxy address.
- `INITIAL_WHITELISTERS` order matters because it's ABI-encoded as `address[]` — same set in different order → different keccak.
- Deployer EOA is the temporary `DEFAULT_ADMIN` of every TLC (constructor arg `address admin`) → encoded into TLC initcode → affects TLC address → affects proxy initcode.
- CREATE2 factory address: Arachnid's `0x4e59…956c` is the chain-of-record singleton. If a chain doesn't have it deployed at that address, the deploy fails. Verified historically on Mainnet, Base, Arb, OP.

## Pre-flight checklist (must pass before broadcast — verify in dry-run phase)

1. **Deployer gas** — `0xb5A6c8ca369e38050784e2A6793beE6447109340` has ETH on each of the 4 chains. Estimate gas budget per chain:
   - Mainnet: ~5M gas total (impl + 3× TLC dance + proxy) at current gwei → record actual estimate from dry-run.
   - L2s (Base/Arb/OP): negligible USD, but confirm balance > 0.01 ETH on each.
   - Worth flagging the [EIP-2470 gas-limit feedback](feedback_eip2470_gas_limit.md): Foundry's auto-estimate has historically under-estimated CREATE2-via-singleton-factory. Pass `--gas-limit 2000000`+ on each script call (production deploys via `forge script --broadcast`).
2. **CREATE2 factory present** — `cast code 0x4e59b44847b379578588920cA78FbF26c0B4956C --rpc-url <chain>` returns non-empty on all 4 chains.
3. **`GOVERNANCE_OWNER` deployed** — Safe multisig `0x59E4DBc95BD312A882Bb36b7f3E8298682340679` confirmed deployed on Mainnet, Base, Arb, OP (verified 2026-05-07: same 344-char bytecode on each — Safe proxy stub). Script's `multisig.code.length == 0` warning branch (line 241-243 of `Deploy.s.sol`) does NOT fire. **Cross-chain identical-address invariant requires identical bytecode at this address on every chain** — verified.
4. **Predicted address parity** — running `PredictAddresses.s.sol` (no broadcast, pure view) against all 4 RPCs MUST return the same proxy address. If not, abort: invariant breach.
5. **No code at predicted address yet** — `cast code <predicted> --rpc-url <chain>` is empty on all 4 chains. (Trivially true if salts haven't been used before, but verify.)
6. **Foundry tests still green** — `forge test -vv` from `contracts/` after the salt bump.

## Risks / open questions

- **Subsquid archive coverage on Mainnet + Arbitrum** — the indexer registry currently lists archives for sepolia / base / base-sepolia / optimism only. Need to verify Subsquid has Ethereum mainnet + Arbitrum archive endpoints (`https://v2.archive.subsquid.io/network/ethereum-mainnet` and `.../arbitrum-one`) before ingestion can start. If not, processor falls back to RPC-only for those chains.
- **Mainnet gas cost** — at typical 1-3 gwei the full deploy is ~$10-30. At 30 gwei it's $100-300. Watch `gasstation` before broadcast.
- **Etherscan API keys** — verification needs per-chain keys. If we don't have an Arbiscan key on file, skip `--verify` for Arb and verify post-hoc.
- **`--broadcast` is a one-way commit** — once broadcast on Base + OP, the v1.1.0 proxy is canonically dead from our perspective even if we don't physically delete it. No rollback. Decision must be irrevocable before tap.
- **Frontend / mesh / indexer cutover atomicity** — the v1.1.0 → v1.2.0 swap on Base + OP isn't atomic. There's a window between broadcast and stack-rewire where the frontend points at v1.1.0 but the new v1.2.0 is live. Mitigation: do the broadcast first, run the indexer briefly against both addresses (or just stop the v1.1.0 indexer), update SM + redeploy services in one wave.
